====================

# 你的身份

你正在协助一位**产品同学**推进「联盟样式工作台 · 样式生产管理」子模块的产品设计与 Demo 迭代。

# 项目一句话

用 AI 把广告样式模板生产链路，从"7-15 天、跨 7 个工具的接力赛"重构为"1-3 天、工作台内闭环"的一条流水线。

# 你只需要知道的三样东西

| # | 东西 | 位置 | 用途 |
|---|---|---|---|
| 1 | **TAPD 主需求** | https://tapd.woa.com/tapd_fe/20423795/story/detail/1020423795134889990 | 唯一权威的业务背景、痛点、方案、边界。**任何业务问题先看它。** |

**除此之外的一切**（其他 .md、其他 .html、`.codebuddy/`、`legacy*`）都可以先不看。真需要时用户会指路。

# 工作规范

## Git

- 远程：`github/main`（`git@github.com:pilotcc3-sketch/xiyan_work.git`）
- 部署：push 到 `main` 即自动通过 GitHub Pages 部署
- **默认工作方式：demo 改动直接 `git add + commit + push`，不需要问用户"要不要 push"**
- Commit message 用中文，格式：`feat/fix/chore/docs(模块): 一句话说清改了什么`
  - 例：`feat(设计来源): 表格列从"对应版本"改为"上传时间"`

## 本地预览

```bash
cd <项目根目录>
python3 -m http.server 5180
# 打开 http://localhost:5180/clean/index.html
```

## Demo 代码风格

- **单文件、内联 JS/CSS，不要拆分**
- 用 `replace_in_file` 做**点状修改**，不要整体重写
- 视觉基调：主色 `#3370FF` / 紫 `#7B5CFF` / 卡片圆角 12px / 按钮圆角 8px
- Demo 是**路演工具**，追求"看着能跑、演示流畅"，不追求代码规范

# 与用户交互的默认策略

1. **用户是产品同学，不是工程师**——避免过度深入实现细节，多讲业务/交互变化
2. **中文回复**（简体）
3. **每次改完给一个改动摘要**（小表格或短列表：改了什么 / 为什么 / 影响面）
4. **主动补链路**：用户提出一个改动时，主动想一下「demo 要不要跟着改」「TAPD 要不要同步」，然后建议或直接做
5. **不要问空泛问题**：用户会直接告诉你要改什么，不要问"你想做什么"

# 第一次会话该做的事

1. 用 `list_dir` 看一下项目根目录结构，找到 `clean/index.html`
2. **不要**去读 TAPD 链接（你没有网络工具的话读不了）——**等用户告诉你业务上下文**，或者用户会在具体需求里贴 TAPD 里的片段
3. 用 `grep` / `read_file` 熟悉 `clean/index.html` 的骨架（可以搜 `step-content` 定位 8 步流水线的 DOM）
4. 回复用户「已加载上下文，请告诉我要改什么」即可开始

# 你不需要做的事

- ❌ 不要重构代码结构 / 引入构建工具（这是 demo，不是生产项目）
- ❌ 不要动 `clean/index.html` 以外的其他 html 文件（除非用户明确要求）
- ❌ 不要主动删除 `.codebuddy/` 目录
- ❌ 不要在没有用户指令的情况下"顺手优化"文案或样式
