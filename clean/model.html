模板分支心智模型

线上永远只有两个版本：主干 L1（模板管理） + 支线 L2（草稿页 · 全局唯一）。提交即合并、合并即覆盖、不可回滚。

① 分支结构 ② 五条硬规则 ③ 主干状态机 ④ 提交合并细节 ⑤ 时间轴 · 实验期间被覆盖 ⑥ 两条主流程 ⑦ 命名约定 ⑧ Demo 改造方案

① 分支结构 · 主干 + 唯一支线

用 git 视角理解：每个 templateId 都有一条主干 L1（永远跑线上的那一版）。任何人在草稿页"开始编辑"那一刻，从当前 L1 fork 出唯一一条支线 L2。L2 走完 ④ 草稿发布 → 立刻物理合并回 L1，支线消失。下一次再编辑，再 fork 一条新 L2。

主干 L1 = 模板管理版本（线上正在跑的） 支线 L2 = 草稿页版本（编辑中、全局唯一） ⚡ = ④ 草稿发布 = 物理合并 + 覆盖
关键差异（与旧"双副本并行"模型对比）：
  • 不再有"两份并行存在的副本"——大多数时刻只有 L1 一条线。L2 只在"有人正在编辑"期间短暂存在。
  • 分叉点从「④ 草稿发布」前移到「点开始编辑」那一刻。
  • L2 是全局单例：所有人共享同一条草稿支线，没有"我的草稿 / 你的草稿"。
  • 合并后 L2 立刻消失，下一轮编辑会基于新 L1 重新 fork。

② 五条硬规则

RULE 1 · 唯一性
草稿页全局唯一
同一 templateId 在任意时刻最多 1 条草稿支线。无论谁来点"开始编辑"，进的都是同一个草稿页、看到的都是同一份 L2 DSL。没有"我的草稿 / 你的草稿"。
RULE 2 · 提交即合并
④ 审批通过 = 物理覆盖
L2 走完 MyOA 三段审批的瞬间，把当前 DSL 快照打包覆盖到 L1，旧 L1 不留底、不可回滚。L2 支线消失，主干回到单线。
RULE 3 · 实验透明
实验/全量只认 templateId
⑤⑥⑦⑧ 都只绑定 templateId，不感知 DSL 变更。所以"⑥ 实验中被覆盖"这种场景，实验照跑、不打断；下一次派量时拿到的就是覆盖后的新 DSL。
RULE 4 · 改动回退
DSL 一动就回 ②
L2 的 DSL 一旦发生任何改动（哪怕一个字符），就自动回退到 ② 编辑态：已通过的测试报告作废、已提的 ④ 审批单作废，必须重跑 ③④。防止"测的是 v1.2、发的是 v1.3"偷渡。
RULE 5 · 共享支线
L2 在 ④ 审批中也允许任意人继续改
如果 L2 已经走到 ④（审批中、未通过），第二个人点"开始编辑"会直接进同一个草稿页继续改，不锁定。但他一改 DSL 就触发 RULE 4 —— 当前审批单作废、回退到 ②。优点：强一致、零额外锁机制；代价：审批中的提单人需要自觉，约定"审批未通过前别再改"。

③ 主干状态机（L1 自身的子状态）

L1 自己也有子状态，对应"在投放上的位置"。L2 没有状态机，永远只有"草稿 · 编辑中"一种状态。

入口：L2 草稿发布 MyOA 三段审批通过 → L2 合并到 L1
↓
⑤
已发布-未生效
L1 进了管理页，但还没配规则 / 还没投放
↓配规则 + 启动 AB 实验
⑥
已发布-实验中
AB 实验进行中，正在收集数据
↓实验结论 OK + MyOA 二段审批（产品/研发）
⑦
已发布-全量
全流量投放，看成绩单
注意：这三个子状态不影响 L2 何时合并。任何子状态下，L2 都可以走 ④ 审批通过、立刻覆盖 L1。覆盖后 L1 子状态被强制重置回 ⑤ 未生效，正在跑的实验/全量不中断（因为只认 templateId），但下一次派量用的就是新 DSL。

④ 提交合并细节 · 草稿发布 = 物理覆盖

合并之前

  • 🚀 L1 主干：v1 · 5 月 28 日
  • 子状态可能是 ⑤ / ⑥ / ⑦ 任意一个
  • 📝 L2 支线：草稿 · 编辑中（基于 v1 fork）
  • L2 已通过 ③ 测试 + ④ MyOA 三段审批
⚡
审批通过

合并之后（瞬间发生）

  • 🚀 L1 主干：v2 · 6 月 10 日（被覆盖）
  • 子状态被强制重置为 ⑤ 已发布-未生效
  • 📝 L2 支线：消失（草稿页清空，等待下一次"开始编辑"）
  • 正在跑的实验/全量：不中断，下次派量取新 DSL
三条不可逆约束：
  1. 把当前 L2 DSL 快照打包，物理覆盖 L1
  2. 旧 L1（v1 · 5/28）不留底、不可回滚
  3. 新 L1（v2 · 6/10）进入 ⑤ 未生效 → 必须重新配规则、重新跑实验（旧的⑤⑥⑦配置作废）

📌 边界：实验/全量不感知 DSL 变更（RULE 3），所以即便覆盖发生在 ⑥ 实验进行中，实验也不会被打断；但实验结论可能因此失效（因为底层 DSL 已变），需要业务方自行判断是否复盘。

⑤ 时间轴 · 实验期间被覆盖（典型剧情）

下面这条时间轴展示一个真实但容易踩坑的场景：L1=v1 正在跑实验（⑥）时，L2 走完 ④，立刻覆盖 L1，实验不中断但底层 DSL 变了。

5/28 6/3 启实验 6/7 fork L2 6/10 ⚡合并 6/12+
🚀 L1 主干
v1 · 5/28 ⑤未生效 → v1 · 6/3 ⑥实验中 → v1 · 6/7 ⑥实验中 → 6/10 被覆盖 → v2 · 6/10 ⑤未生效（实验仍在跑但底层是 v2）
📝 L2 支线
（无支线） → （无支线） → 6/7 fork · ②编辑 → 6/9 ③测试 + ④审批 → （合并消失）
关键观察：
  • 5/28 ~ 6/6：L2 不存在，主干上只有 v1 一条线在跑实验。
  • 6/7：有人点"开始编辑" → 从 v1 fork 出 L2。两条线并存（仅这几天）。
  • 6/10：L2 ④ 审批通过 → L2 物理覆盖 L1（v1→v2），L2 立刻消失。
  • 6/10 之后：实验依然在跑（只认 templateId），但实验组从下一次派量开始拿到的就是 v2 DSL。实验数据从此变得"前 v1 后 v2"，需要业务自行判断结论是否还有效。
  • L1 子状态被重置回 ⑤ 未生效，意味着原本 ⑤⑥⑦ 的配置作废，需要重新配规则才能让 v2 进入新的 ⑥ 实验。

⑥ 两条主流程

流程 A · 新模板首次上线

① 创建（L2 from 空） → ② 编辑 DSL → ③ 测试 → ⚡ ④ 草稿发布 L2 支线
↓ 合并 · L1 首次诞生
⑤ 已发布-未生效 → ⑥ 已发布-实验中 → ⑦ 已发布-全量 → ⑧ 成绩单 L1 主干

流程 B · 老模板迭代（高频循环）

L1 = v1（⑤/⑥/⑦ 任一状态） L1 主干
↓ 任意时刻有人"开始编辑" · fork L2
② 编辑下一版 → ③ 测试 → ⚡ ④ 草稿发布 L2 支线（基于 v1）
↓ 合并 · 物理覆盖（v1 死掉）
L1 = v2 · ⑤未生效 → ⑥ 重新跑实验 → ⑦ v2 全量 L1 主干（被覆盖后）
↻ 周而复始，每次"开始编辑→④发布"都覆盖一次 L1

⑧ Demo 该怎么改

当前 demo 把 8 步流水线画成一条直线，掩盖了"主干 + 临时支线 + 提交即合并"的真相。下面 5 个改造点，按"骨架→交互→警示→列表→抽屉"由外到内列出。

改造点 ① 顶部 8 步骨架图：从「直线串行」改为「分支视图」

✗ 现状：直线串行
①创建 → ②编辑 → ③测试 → ④草稿发布 → ⑤规则 → ⑥实验 → ⑦全量 → ⑧成绩单
问题：①②③④ 是支线、⑤⑥⑦⑧ 是主干，并不是同一条线；④ 后还会循环回 ②。
