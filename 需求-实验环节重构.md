# 需求 · 模板实验环节重构（⑥ 模板实验 → 必经化）

> 模板生产流水线 · 后台状态机与审批流重构
> 提单人：小樊 · 2026-06-25

> 🔑 **本质说明**：本需求是**样式生产管理后台**的状态机 + 审批流重构。新平台（一站式工作台）和旧平台（`o.adnet.qq.com/sketch/...`）**共用同一套后端逻辑**，差异仅在前端外显方式。所以本文档以"后台逻辑"为主线，新旧平台的 UI 改造均作为同一逻辑的不同展示层附属说明。

---

## 一、需求背景

当前模板生产流水线的状态流转中，「已发布-实验中」**不是**从「已发布-未生效」到「已发布-全量」的必经状态——
- 它和「已发布-全量」**共享同一条 MyOA 审批流**，审批通过即可任选其一
- 想真正跑实验，还要**额外**去 TAB 实验平台再配一个实验
- 两者都做完，实验才会真正生效

这导致以下问题：

| # | 问题 | 影响 |
|---|---|---|
| 1 | **无审批可绕过实验**：模板可以从「未生效」直接审批到「全量」，跳过实验环节，无任何卡点 | 大量模板未经 AB 验证直接全量，效果回流困难、风险无法事前评估 |
| 2 | **审批流职责混淆**：同一个审批单既可能是"上实验"也可能是"上全量"，审批人无法清晰判断本次审批的影响范围 | 审批人决策成本高、错批风险大 |
| 3 | **TAB 配置游离在流水线外**：实验配置在 TAB 平台单独完成，与样式生产管理后台两套数据，状态不同步 | 工作台无法准确反映模板的真实实验状态 |
| 4 | **跳过实验无审计**：无实验直接全量的模板，没有任何记录说明"为什么不做实验" | 复盘困难、合规审计无依据 |

---

## 二、需求价值

1. **质量收口**：所有模板**默认必须经过实验验证**，效果数据闭环，全量决策有据可依
2. **审批职责清晰**：实验上线 / 跳过实验 / 全量上线 拆成 3 个独立审批流，各自承担明确职责
3. **状态机收敛**：「已发布-实验中」从"可选分支"变为"必经节点"，流水线状态机更可预测
4. **异常通道留口**：通过「跳过实验审批流」保留紧急/无需实验的逃生通道，不一刀切
5. **新旧平台一致**：后端共用一套，新旧平台对同一模板的状态显示永远一致，杜绝"在 A 平台看到的状态和 B 平台不一样"

## 四、需求描述（后台逻辑）

### 4.1 状态机变化（核心 · 后台）

**【旧】**

```
已发布-未生效
    │
    ├──(MyOA 实验/全量 合并审批通过)──> 已发布-实验中 *
    │
    └──(MyOA 实验/全量 合并审批通过)──> 已发布-全量
                                          ↑
                                          │
* 「已发布-实验中」需额外去 TAB 平台配实验才真正生效；与「已发布-全量」并列、可绕过
```

**【新】**

```
已发布-未生效
    │
    ├──[主路径]──> 提交「实验上线」MyOA 审批 ─产研两轮─> 已发布-实验中 ─> 跳转 TAB 平台配实验
    │                                                      │
    │                                                      └─ 实验跑完、达到决策条件
    │                                                            ↓
    │                                                      提交「全量上线」MyOA 审批 ─> 已发布-全量
    │
    └──[异常分支]──> 提交「跳过实验」MyOA 审批 ─产研两轮─> 已发布-未生效（标记"已批准跳过实验"）
                                                            ↓
                                                      提交「全量上线」MyOA 审批 ─> 已发布-全量
```

**关键变化**：
1. 「已发布-实验中」成为**主路径必经节点**
2. 「跳过实验」拆为**独立审批流**，通过后状态保持「已发布-未生效」但带"已批准跳过实验"标记
3. 「全量上线」**永远是独立审批流**，且前置条件必须满足"实验通过" 或 "跳过实验通过"
4. 全量上线提单接口的服务端校验：`exp_state ∈ {published, skipped}`，**任何前端入口都绕不过**

### 4.2 审批流拆分

| # | 审批流名称 | 触发动作 | 审批人 | 通过后状态 |
|---|---|---|---|---|
| 1 | **实验上线**（新拆出，旧"实验/全量合并"的一半） | 调用 `POST /sketch/exp/submit` | 产品 2 人 → 研发 3 人 | exp_state = `published`（已发布-实验中） |
| 2 | **跳过实验**（全新） | 调用 `POST /sketch/exp/skip-submit` | 产品 2 人 → 研发 3 人 | exp_state = `skipped`（已批准跳过实验） |
| 3 | **全量上线**（新拆出，旧"实验/全量合并"的另一半） | 调用 `POST /sketch/full/submit` | 产品 2 人 → 研发 3 人 | full_state = `published`（已发布-全量） |

**服务端前置校验（任何来源平台都生效）**：
- `/sketch/full/submit` 提单时校验 `exp_state ∈ {published, skipped}`，否则返回 `4001 EXP_NOT_PASSED`，错误信息体里附引导文案
- 审批流 #1、#2 **互斥**：任一通过后，另一个的提交接口返回 `4002 EXP_ALREADY_DECIDED`（前端据此禁用入口）；已通过方可调"撤销"接口重新走另一条

### 4.3 后端数据模型变化

```sql
-- 模板状态表新增 2 个字段
ALTER TABLE template_state ADD COLUMN exp_state ENUM('idle','pending','published','skipped') DEFAULT 'idle';
ALTER TABLE template_state ADD COLUMN legacy_skip_exp TINYINT(1) DEFAULT 0;  -- 存量迁移标记

-- 新增审计表（新旧平台共用）
CREATE TABLE sketch_approval_audit (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  tpl_id BIGINT NOT NULL,
  action ENUM('exp_publish','exp_skip','full_publish') NOT NULL,
  reason TEXT,                        -- 跳过实验时必填
  reason_category VARCHAR(32),        -- 跳过原因类别
  approver_chain JSON,                -- 审批人链路快照
  source_platform ENUM('new','legacy') NOT NULL,  -- 来源平台，仅用于统计，不影响逻辑
  created_at DATETIME NOT NULL,
  INDEX idx_tpl_action (tpl_id, action)
);
```

### 4.4 拦截规则（服务端强制 + 前端引导双层）

> 🔑 **核心原则**：服务端是**唯一权威**，所有拦截规则都在服务端实现。前端（无论新旧）只是把服务端返回的错误码翻译成更友好的引导文案。

| 拦截编号 | 触发动作 | 服务端校验 | 错误码 | 前端引导文案（参考） |
|---|---|---|---|---|
| **B1** | 调 `/sketch/full/submit` | `exp_state ∈ {idle, pending}` | `4001 EXP_NOT_PASSED` | "全量上线前需先完成实验环节，请选择：① 发起实验上线审批 / ② 发起跳过实验审批" |
| **B2** | 调 `/sketch/full/submit` | `exp_state = published` 但实验时长 < 7 天 | `2001 EXP_DURATION_WARN`（非阻塞，仅警告） | "实验观察期未满 7 天，是否继续？" |
| **B3** | 调 `/sketch/exp/skip-submit` | `reason` 字段为空 | `4003 SKIP_REASON_REQUIRED` | "跳过实验需说明原因（必填）+ 类别下拉" |
| **B4** | 实验审批通过后 24h，TAB 平台仍无实验配置 | 后台轮询 + 企微提醒 | 无 HTTP 错误，走企微推送 | "已通过实验上线审批，但 TAB 平台尚未配置实验" |
| **B5** | 模板 `exp_state = idle` 持续 > 7 天 | 后台扫描 + 企微提醒提单人 | 无 HTTP 错误 | "该模板已发布 7 天未进入实验" |

> ⚠️ B1 / B3 是**硬拦截**（服务端 4xx），B2 是**软警告**（服务端 2xx + 提示码），B4 / B5 是**异步推送**（不阻塞用户操作）。

### 4.5 存量数据迁移（一次性脚本）

```
UPDATE template_state
SET legacy_skip_exp = 1, exp_state = 'skipped'
WHERE full_state = 'published' AND exp_state = 'idle';
```

迁移完后，所有历史"已发布-全量"模板都被视作"已批准跳过实验"，⑦/⑧ 节点继续可访问，前端在抽屉里展示一个灰色 badge：`历史模板 · 自动迁移`。

### 5.3 用户引导文案（新旧平台共用文案库）

后端在错误码体里返回**结构化提示**，前端按各自 UI 风格渲染。文案库统一维护一份：

| 错误码 | 拦截类型 | 推荐 UI 形态（新平台） | 推荐 UI 形态（旧平台） | 文案 |
|---|---|---|---|---|
| `4001 EXP_NOT_PASSED` | 硬拦截 | Modal 弹窗 + A/B 两个出路按钮 | Modal 弹窗 + A/B 两个出路按钮 | "全量上线前需先完成实验环节。请选择：①发起实验上线审批 / ②发起跳过实验审批" |
| `2001 EXP_DURATION_WARN` | 软警告 | Confirm 弹窗（继续/再等等） | Confirm 弹窗（继续/再等等） | "实验观察期未满 7 天，是否继续？" |
| `4003 SKIP_REASON_REQUIRED` | 硬拦截 | Form 弹窗（原因必填 + 类别下拉） | Form 弹窗（原因必填 + 类别下拉） | "跳过实验需说明原因。类别：紧急修复 / 无 AB 价值 / 历史补流程 / 其他" |
| B4 异步推送 | 提醒 | 顶部黄条 + 企微 | 顶部黄条 + 企微 | "已通过实验上线审批，但 TAB 平台尚未配置实验" |
| B5 异步推送 | 提醒 | 流水线节点红点 + Tooltip | 列表行红点 + Tooltip | "该模板已发布 7 天未进入实验" |

**统一规范**：
- **拒绝必给出路**——硬拦截弹窗必须提供 2 个或以上行动按钮
- **"实验环节" 四个字一律可点击**——跳转到 ⑥ 模板实验（新旧平台各自的对应页面）
- **决策树文档**统一落在 KM：`/km/样式生产管理/实验环节必经制度.md`，新旧平台帮助入口都指向这里

## 六、需要特别关注

### 6.1 客服周知
**不涉及**——本需求是站内后台状态机重构，不影响 SDK / 投放 / 外网。

### 6.2 上线后数据验证
- **核心指标**：上线 30 天内，"经过实验通过后全量"的模板占比 ≥ 80%（基线：当前约 30%）
- **跳过实验占比**：≤ 20%，超阈值需复盘
- **审批时长**：实验上线审批 P90 ≤ 24h，跳过实验审批 P90 ≤ 12h
- **新旧平台一致性**：抽样 50 个模板，新旧平台显示状态 100% 一致

### 6.3 风险点

| 风险 | 影响 | 缓解 |
|---|---|---|
| **存量"已发布-全量"模板**已绕过实验环节，无历史实验数据 | 不影响——存量按"已批准跳过实验"语义自动迁移 | 一次性数据迁移脚本，给所有存量"已发布-全量"模板补标记 `legacy_skip_exp = 1` |
| **跳过实验滥用**——产研把所有模板都走跳过流程，等于没改 | 失去本次重构价值 | 跳过实验审批人 + 产品 + 研发，两轮卡控；跳过率作为月度复盘指标，超 20% 触发复盘 |
| **TAB 平台配置游离**——审批通过了但研发没去 TAB 配实验 | 实验未真正跑起来 | B4 异步推送：24h 无数据则企微提醒提单人 |
| **历史 in-flight 模板**——重构上线时刚好处于"实验/全量合并审批中"状态 | 审批通过后不知道走哪条路径 | 上线前 1 周冻结合并审批入口；存量审批单按"实验上线"语义处理 |
| **新旧平台不同步上线** | 用户在旧平台依然能绕过实验 | 服务端硬拦截前置——只要旧平台调的是新接口，逻辑就一定生效；接口收口必须先于规则切换 |

## 七、发布与验收

### 7.1 发布顺序（关键）

```
D-14 天：后端接口部署（双写：新接口可用，旧接口仍兼容）
D-7  天：旧平台前端切换调新接口（此时规则尚未启用，仅打通链路）
D-3  天：群发文档 + 30s 视频 demo，旧平台首页挂公告条
D-Day：后端开启拦截校验（新旧平台同步生效，无需各自发版）
D+7  天：收集第一周拦截事件 Top10，调整文案
D+30 天：移除公告条；下线旧接口
```

> 🔑 **重点**：因为新旧平台共用同一套服务端，所以 D-Day 当天**只需切换服务端开关**，新旧平台同时生效，不存在"旧平台先漏一段时间"的窗口期。

### 7.2 发布方式
- **站内后台**全量发布，无需灰度
- 不影响线上投放、SDK、外部业务

### 7.3 验收方式
