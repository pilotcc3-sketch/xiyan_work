# 需求 · 模板实验环节重构（⑥ 模板实验 → 必经化）

> 模板生产流水线 · ⑥ 模板实验环节状态机重构
> 提单人：小樊 · 2026-06-25

---

## 一、需求背景

当前模板生产流水线的状态流转中，「已发布-实验中」**不是**从「已发布-未生效」到「已发布-全量」的必经状态——
- 它和「已发布-全量」**共享同一条 MyOA 审批流**，审批通过即可任选其一
- 想真正跑实验，还要**额外**去 TAB 实验平台再配一个实验
- 两者都做完，实验才会真正生效

这导致以下问题：

| # | 问题 | 影响 |
|---|---|---|
| 1 | **无审批可绕过实验**：模板可以从「未生效」直接审批到「全量」，跳过实验环节，无任何卡点 | 大量模板未经 AB 验证直接全量，效果回流困难、风险无法事前评估 |
| 2 | **审批流职责混淆**：同一个审批单既可能是"上实验"也可能是"上全量"，审批人无法清晰判断本次审批的影响范围 | 审批人决策成本高、错批风险大 |
| 3 | **TAB 配置游离在流水线外**：实验配置在 TAB 平台单独完成，与样式生产管理后台两套数据，状态不同步 | 工作台无法准确反映模板的真实实验状态 |
| 4 | **跳过实验无审计**：无实验直接全量的模板，没有任何记录说明"为什么不做实验" | 复盘困难、合规审计无依据 |

---

## 二、需求价值

1. **质量收口**：所有模板**默认必须经过实验验证**，效果数据闭环，全量决策有据可依
2. **审批职责清晰**：实验上线 / 跳过实验 / 全量上线 拆成 3 个独立审批流，各自承担明确职责
3. **状态机收敛**：「已发布-实验中」从"可选分支"变为"必经节点"，流水线状态机更可预测
4. **异常通道留口**：通过「跳过实验审批流」保留紧急/无需实验的逃生通道，不一刀切

## 四、需求描述

### 4.1 状态机变化（核心）

**【旧】**

```
已发布-未生效
    │
    ├──(MyOA 实验/全量 合并审批通过)──> 已发布-实验中 *
    │
    └──(MyOA 实验/全量 合并审批通过)──> 已发布-全量
                                          ↑
                                          │
* 「已发布-实验中」需额外去 TAB 平台配实验才真正生效；与「已发布-全量」并列、可绕过
```

**【新】**

```
已发布-未生效
    │
    ├──[主路径]──> 提交「实验上线」MyOA 审批 ─产研两轮─> 已发布-实验中 ─> 跳转 TAB 平台配实验
    │                                                      │
    │                                                      └─ 实验跑完、达到决策条件
    │                                                            ↓
    │                                                      提交「全量上线」MyOA 审批 ─> 已发布-全量
    │
    └──[异常分支]──> 提交「跳过实验」MyOA 审批 ─产研两轮─> 已发布-未生效（标记"已批准跳过实验"）
                                                            ↓
                                                      提交「全量上线」MyOA 审批 ─> 已发布-全量
```

**关键变化**：
1. 「已发布-实验中」成为**主路径必经节点**
2. 「跳过实验」拆为**独立审批流**，通过后状态保持「已发布-未生效」但带"已批准跳过实验"标记
3. 「全量上线」**永远是独立审批流**，且前置条件必须满足"实验通过" 或 "跳过实验通过"
4. ⑦ / ⑧ 节点不可点击的拦截：检查 ⑥ 的 `data-exp-state ∈ {published, skipped}`

### 4.2 审批流拆分

| # | 审批流名称 | 触发节点 | 审批人 | 通过后状态 |
|---|---|---|---|---|
| 1 | **实验上线**（新拆出，旧"实验/全量合并"的一半） | ⑥ A 区"提交实验上线" | 产品 2 人 → 研发 3 人 | data-exp-state = `published`（已发布-实验中） |
| 2 | **跳过实验**（全新） | ⑥ D 区"申请跳过实验" | 产品 2 人 → 研发 3 人 | data-exp-state = `skipped`（已批准跳过实验） |
| 3 | **全量上线**（新拆出，旧"实验/全量合并"的另一半） | ⑦"提交全量上线" | 产品 2 人 → 研发 3 人 | data-full-state = `published`（已发布-全量） |

**前置校验**：
- 审批流 #3 提单时，校验 data-exp-state ∈ {published, skipped}，否则禁止提交
- 审批流 #1、#2 互斥：任一通过后，另一个的提交入口禁用（已通过的一方可"撤销"重新走另一条）

### 4.3 UI 变化（基于现有 demo · clean/index.html）

⑥ 模板实验节点页面分 4 个卡片，**已在 demo 中落地**：

| 区块 | 内容 | 状态 |
|---|---|---|
| A | 📨 实验上线审批（产研两轮 MyOA） | ✅ Demo 已实现 |
| B | 🧪 跳转 TAB 实验平台 | ✅ Demo 已实现 |
| C | 🟢 模板在线 adata 数据卡 | ✅ Demo 已实现 |
| D | ↷ 申请跳过实验（异常分支 · 与 A 二选一） | ✅ Demo 已实现 |

**首页过滤 pill 不变**：保留「全部 / 已发布-未生效 / 已发布-实验中 / 已发布-全量」，无需新增过滤项（"已批准跳过实验" 仍归在「已发布-未生效」下，仅在抽屉里以标记形式呈现）。

**优化前 vs 优化后截图对比**：见 demo（https://pilotcc3-sketch.github.io/xiyan_work/） · 点任意模板「流水线 →」→ ⑥ 模板实验。

### 4.4 流水线节点拦截规则（已在 demo 中实现）

- 点击 ⑦ / ⑧ 节点时校验 `data-exp-state`：
  - `idle` / `pending` → 拦截，提示"⑥ 实验尚未审批通过"
  - `published` / `skipped` → 放行

## 五、需要特别关注

### 5.1 客服周知
**不涉及**——本需求是站内后台状态机重构，不影响 SDK / 投放 / 外网。

### 5.2 上线后数据验证
- **核心指标**：上线 30 天内，"经过实验通过后全量"的模板占比 ≥ 80%（基线：当前约 30%）
- **跳过实验占比**：≤ 20%，超阈值需复盘
- **审批时长**：实验上线审批 P90 ≤ 24h，跳过实验审批 P90 ≤ 12h

### 5.3 风险点

| 风险 | 影响 | 缓解 |
|---|---|---|
| **存量"已发布-全量"模板**已绕过实验环节，无历史实验数据 | 不影响——存量按"已批准跳过实验"语义自动迁移 | 一次性数据迁移脚本，给所有存量"已发布-全量"模板补标记 `legacy_skip_exp = true` |
| **跳过实验滥用**——产研把所有模板都走跳过流程，等于没改 | 失去本次重构价值 | 跳过实验审批人 + 产品 + 研发，两轮卡控；跳过率作为月度复盘指标 |
| **TAB 平台配置游离**——审批通过了但研发没去 TAB 配实验 | 实验未真正跑起来 | ⑥ C 区"在线数据"如果 24h 后无数据，自动企微提醒提单人 |
| **历史 in-flight 模板**——重构上线时刚好处于"实验/全量合并审批中"状态 | 审批通过后不知道走哪条路径 | 上线前 1 周冻结合并审批入口；存量审批单按"实验上线"语义处理 |

## 六、发布与验收

### 6.1 发布方式
- **站内后台**全量发布，无需灰度
- 配合 MyOA 审批流上线时间，提前 1 周冻结旧合并审批入口
- 老审批单（in-flight）按"实验上线"语义处理，不强制重审

### 6.2 验收方式

| # | 验收场景 | 预期 |
|---|---|---|
| 1 | 新建模板 → ④ 草稿发布通过 → ⑤ 规则配置 → ⑥ A 区提交实验上线 → 审批通过 | 状态变为「已发布-实验中」，⑦ 全量上线节点解锁 |
| 2 | 新建模板 → 走到 ⑥ → D 区申请跳过实验 → 审批通过 | 状态保持「已发布-未生效」+ "已批准跳过实验" 标记，⑦ 解锁 |
| 3 | 新建模板 → 走到 ⑥ → 同时点 A 和 D | A 提交后 D 入口禁用 / D 提交后 A 入口禁用 |
| 4 | ⑥ 未审批 → 直接点 ⑦ 或 ⑧ | 弹窗拦截，提示"⑥ 模板实验尚未审批通过" |
| 5 | 存量「已发布-全量」模板 | 自动补 `legacy_skip_exp = true` 标记，⑦ / ⑧ 正常可看 |
| 6 | 已通过实验上线 → 想改 DSL | ⑥ 顶部出现"重新提交实验上线"提醒 banner |

### 6.3 影响范围

| 模块 | 是否影响 |
|---|---|
| 样式生产管理后台 | ✅ 状态机 + 审批流重构 |
| MyOA | ✅ 新增 1 条审批流 + 原合并流拆分为 2 条 |
| TAB 实验平台 | ❌ 无改造，仅作为外部跳转目标 |
| adata 数据看板 | ❌ 无改造 |
| SDK（标准/聚合/数据版/Unity） | ❌ 不涉及 |
| 投放、计费、客服 | ❌ 不涉及 |

### 6.4 统计需求

需新增以下事件上报：

| 事件名 | 触发时机 | 字段 |
|---|---|---|
| `tpl_exp_submit` | ⑥ A 区提交实验上线 | tplId, submitter |
| `tpl_exp_skip_submit` | ⑥ D 区提交跳过实验 | tplId, submitter, reason |
| `tpl_exp_approve_pass` | ⑥ 实验上线审批通过 | tplId, approvers, duration_h |
| `tpl_exp_skip_approve_pass` | ⑥ 跳过实验审批通过 | tplId, approvers, duration_h |
| `tpl_full_submit_blocked` | ⑦ 提单时被前置校验拦截 | tplId, reason |

## 七、用户引导与拦截文案（新/旧平台都需实现）

新规则下，用户最容易踩坑的两个动作是：① 没经过实验直接点「全量上线」；② 想做实验但不知道现在要先发起「实验上线审批」。必须在 UI 上给出**明确的拦截 + 指引**，否则会引发大量客诉。

### 7.1 关键拦截场景（共 5 个）

| # | 触发动作 | 当前状态 | 拦截/引导文案 | UI 形态 |
|---|---|---|---|---|
| **L1** | 点击「发起全量上线审批」 | `exp-state = idle` 或 `pending`（未经过实验，也未跳过） | **"全量上线前需先完成实验环节"**<br>请先在 ⑥ 模板实验中选择：<br>· **A. 发起实验上线审批** → 审批通过后在 TAB 配实验、观察 ≥ 7 天<br>· **B. 发起跳过实验审批**（仅用于无需实验的场景，需说明原因）<br>[去 ⑥ 模板实验] | **Modal 弹窗**（不可关闭，必须二选一） |
| **L2** | 点击「发起全量上线审批」 | `exp-state = published` 但实验时长 < 7 天 | **"实验观察期未满 7 天，是否继续？"**<br>当前已观察 X 天，建议至少 7 天再发起全量。如确需提前全量，请在审批单中填写原因。<br>[继续发起] [再等等] | **Confirm 弹窗**（可继续，但留痕） |
| **L3** | 点击「发起跳过实验审批」 | 任意 | **"跳过实验需说明原因（必填）"**<br>· 紧急修复 bug，无实验空间<br>· 模板逻辑无 AB 价值（如纯文案修改）<br>· 历史模板补流程<br>· 其他（请详述）<br>[提交审批] | **Form 弹窗**（必填原因 + 类别下拉） |
| **L4** | 在 ⑥ 实验环节看到「在线数据」区为空 | `exp-state = published` 但 TAB 侧未配实验 | **"已通过实验上线审批，但 TAB 平台尚未配置实验"**<br>请在 30 分钟内完成 TAB 实验配置，否则实验状态将自动回退。<br>[去 TAB 配置 ↗] [查看配置指引] | **顶部黄条 Warning** |
| **L5** | 进入流水线页面，模板状态为「已发布-未生效 > 7 天」 | `exp-state = idle` 持续 > 7 天 | **"该模板已发布 7 天未进入实验"**<br>建议尽快推进至实验环节，避免占用产线资源。<br>[发起实验] [发起跳过实验] [暂不处理] | **流水线节点 Badge 红点 + Tooltip** |

### 7.2 文案规范

- **拒绝式**（L1）必须给"出路"——同时提供 A/B 两个按钮，不能只说"不行"
- **警告式**（L2/L4）允许通过，但要让用户**意识到自己在做什么**
- **引导式**（L5）不打断用户，靠红点 + Tooltip 提醒
- **统一口径**：所有文案里出现的"实验环节"四个字必须可点击，跳到 ⑥ 模板实验

### 7.3 帮助入口

- 在 ⑥ 模板实验区头部加一个**「为什么实验环节必经？」**的浮窗（hover/click），里面放：
  - 新旧流程对比图（直接复用本文档第四节的状态机图）
  - "我应该选实验还是跳过？" 决策树
  - 常见问题 FAQ（5 条）
- 文档落在 KM：`/km/样式生产管理/实验环节必经制度.md`（待新建）

## 八、旧平台同步改造（不能只改新平台）

> ⚠️ **重要**：当前仍有大量同事/外部业务方在使用**旧版样式生产管理后台**（站内地址 `o.adnet.qq.com/sketch/...`），如果只改新平台，旧平台依然能"未经实验直接全量"，规则就形同虚设。新旧必须**同步上线、同步生效**。

### 8.1 新旧平台差异盘点

| 模块 | 旧平台现状 | 新平台目标 | 旧平台改造动作 |
|---|---|---|---|
| **状态机** | 「已发布-实验中」是可选分支 | 「已发布-实验中」是必经节点 | 同步状态机改造 |
| **审批流** | 1 条合并审批流 | 拆为 3 条（实验/跳过/全量） | 同步拆分，且**与新平台共用同一套 MyOA 模板** |
| **全量入口拦截** | 无拦截，直接可点 | L1 ~ L5 五处拦截 | 同步实现 L1 / L3 / L5（L2/L4 视旧平台 UI 容量决定） |
| **跳过实验入口** | 不存在 | 显式按钮 + 必填原因 | 新增同样按钮 |
| **审计日志** | 无 | 记录跳过原因、审批人、时间 | 同步落库（**与新平台同一张表**） |
| **历史数据** | 大量"未经实验直接全量"的存量 | `legacy_skip_exp = true` 标记 | 数据迁移脚本**新旧共用**，一次性处理 |

### 8.2 旧平台改造范围（按优先级）

#### P0 · 必须改（上线日 D-Day 同步生效）
1. **拦截 L1**：旧平台「全量上线」按钮点击时，校验 `exp-state`，未经过实验则弹窗拦截
2. **拆审批流**：旧平台调用 MyOA 的接口同步切换到新的 3 条审批流模板 ID
3. **跳过实验入口**：在旧平台流水线页加一个「跳过实验」按钮（位置：实验环节卡片右上角）
4. **审计日志共表**：旧平台跳过/全量审批通过后，写入与新平台**同一张** `sketch_approval_audit` 表

#### P1 · 强烈建议（D-Day + 2 周内补）
5. **拦截 L5**：流水线列表页加红点提醒
6. **拦截 L3**：跳过实验原因必填 + 类别下拉
7. **帮助浮窗**：FAQ + 决策树（可只放静态文案，不做交互）

#### P2 · 可选（视旧平台维护成本）
8. **拦截 L2** / **L4**：旧平台 UI 改造成本若 > 2 人日，可降级为"提交审批时在表单里加文字提示"
9. **新旧平台数据双向同步**：用户在旧平台发起的审批，新平台流水线节点状态也要联动变化（如已经是这样可以不动；如不是，建议直接对接同一份后端数据源，而非两套 mock）

### 8.3 旧平台改造方法选择

**强烈推荐方案 A：复用新平台后端**
- 旧平台前端不动，只改"点击全量上线按钮"和"跳过实验按钮"的**接口调用**，统一打到新平台后端
- 后端是同一套，状态机、审批流、审计日志、数据迁移都不会出现新旧不一致
- 改造成本：旧平台 ≈ 3 人日，主要是接口替换 + 弹窗组件

**方案 B：旧平台单独实现一套**（不推荐）
- 旧平台维护一套自己的状态机和审批流配置
- 改造成本：≈ 10 人日，且未来每次新平台改规则，旧平台都要跟改一遍
- 唯一适用场景：旧平台后端与新平台彻底隔离、短期无法打通

> **决策**：采用方案 A。配合本次重构，把旧平台"全量上线/跳过实验"两个动作的后端接口收口到新平台，作为后续整体下线旧平台的第一步。

### 8.4 旧平台过渡期沟通

| 时间点 | 动作 | 责任人 |
|---|---|---|
